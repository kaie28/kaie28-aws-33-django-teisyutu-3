### 1.【公開】

・AWSとDjangoのソースコードは、このGitHubに公開しています。

・webアプリ環境は、現在閉鎖中です。

### 2【概要】

・Ubuntuでの環境構築からシンプルなTODOアプリ作成を通じ、TerraformやDockerを用いたインフラ自動化とセキュリティ対策を実践しました。この経験を土台に、オンプレミスクラウドの領域まで幅広く学び、多様な案件に対応できる技術力を身につけたいと考えています。

・AWS上では22番ポートを外部公開せず、SSMと連携させたセキュアなSSH接続モデルを設計・実装しました。この「22番を晒さない」セキュリティ思想を、ネットワークにおける踏み台構成やアクセス制御の設計構築にも活かしていきます。


### 3【開発背景】

・日々愛用している「Instagram」の裏側を支える大規模なインフラ構造に興味を持ち、サーバー構築やネットワークの基礎から学び始めました。今後はシステム移行や、さらなる自動化（IaC）による運用負荷の削減、新規構築にも挑戦していきたいと考えています。

・未経験から5ヶ月での完成を目指し、計画的に学習と開発を進めました。特に「セキュリティ強化」「迅速なデータ復元」「コスト最適化」にこだわり、設計から構築までの全工程を独学で実装しました。



### 4【工夫した点】

・本番環境においてはDockerを用いたコンテナ運用を行い、同時に、本番稼働中であってもリアルタイムでログからエラーを追跡・検知できる監視設計を組み込みました。

・シークレットキー等の接続情報などの機密事項を.env および .gitignore を用いて環境変数として外部管理することで、GitHub経由の認証情報漏洩リスク対策をしました。

・当初の「手動構築」におけるデータ消失の失敗経験を機に、環境をすぐ再現できる仕組みの確保が不可欠だと痛感しました。そこで、再構築のために「Docker」でのコンテナ化と「Terraform」による自動化を採用し、復元しやすい環境へと強化しました。

【DBの注意】開発/本番環境は共に「db.sqlite3」を使用していますが、本番環境ではDockerのボリューム機能(コンテナ外の安全金庫)である「db_volume」と紐づけています。なので、コンテナの再起動や予期せぬ停止が発生しても「文字・CSS・画像」は消えず、安全に保持されます。(※計画時の目標に、「WebサーバーとDjango連携を最速で学習する」を優先したため、この形式にしました)。
